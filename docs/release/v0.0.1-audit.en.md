# v0.0.1 Release Audit

[中文版本](./v0.0.1-audit.md)

This document records the current `v0.0.1` release audit state. Its purpose is to separate items backed by strong evidence from items that still require another dedicated pass, instead of treating a successful build as proof that the whole release is done.

## Audit Scope

This pass covers only:

1. Whether the `v0.0.1` version is aligned everywhere
2. Whether the Android-first MVP has baseline release evidence
3. Whether the local Android debug / release automation paths actually run
4. Whether the publishing page / Vercel assets are in-repo and buildable
5. Which requirements are still missing

## Objective Mapping

| Objective Item | Status | Evidence |
| --- | --- | --- |
| Baseline MVP runs | Strong evidence | `npm run test -- --run`, `npm run typecheck`, `npm run export:android` |
| Design is prepared | Partially complete | Design and execution plans exist under `docs/plans/2026-04-16-android-media-cleaner-*`; publishing assets are in repo |
| Android product completeness check | Not complete | Current evidence is test-level plus local build evidence; interactive acceptance is still missing |
| Android automation flow check | Partially complete | Local release/debug scripts were executed successfully; GitHub Actions still lack remote run evidence |

## Current Product Completeness Evidence

Current "product completeness" evidence is still mostly test-level, not final interaction-level acceptance:

1. Scan and progress: `src/features/scan/__tests__/scan-with-progress.test.ts`
2. Recycle bin and candidate state: `src/application/media-cleaner-helpers.test.ts`, `src/ui/screens/__tests__/RecycleBinScreen.test.tsx`
3. Settings for scan range, language, and theme: `src/ui/screens/__tests__/SettingsScreen.test.tsx`
4. Local reminders and scheduling: `src/services/notifications/cleanup-reminders.test.ts`
5. Scan completion notifications: `src/services/notifications/scan-completion-notifications.test.ts`
6. Notch / hole-punch / foldable / tablet adaptation: `src/features/compatibility/__tests__/screen-adaptation.test.ts`

These tests show that the main logic and adaptation rules are encoded and protected, but they do not replace real Android interaction acceptance.

## Current Gaps

The following items still cannot be marked complete:

### 1. Android Product Completeness Still Lacks Real Acceptance

The following interaction-level checks are still missing:

1. First-run permission request and denied-permission recovery
2. The scan primary flow and result-list interactions
3. Auto cleanup / manual cleanup / recycle-bin restore
4. Reminder-setting changes and their UI-to-notification behavior
5. Language/theme switches across actual screens
6. Visual checks on notch, hole-punch, and foldable devices

### 2. GitHub Actions Still Lack Remote Evidence

The repository already includes:

1. `.github/workflows/android-release.yml`
2. `.github/workflows/android-debug.yml`
3. `.github/workflows/android-pr-check.yml`

But this pass did not obtain:

1. Real successful runs on GitHub runners
2. `actionlint` or equivalent workflow syntax validation

So the current conclusion can only be:

> the workflows exist and their local equivalent scripts pass, but the remote automation loop is not fully proven yet.

## Conclusion

At the current audit point:

1. The `v0.0.1` baseline MVP, publishing assets, and local Android debug / release packaging paths all have strong evidence
2. `Android product completeness` and `GitHub Actions remote automation closure` are still not fully accepted
3. Therefore the correct status statement is:

> `v0.0.1` already has the local release foundation and audit evidence, but the overall release objective is not complete yet and still needs another focused session.

## Next Pass Recommendation

1. Run a real-device or Android-emulator product acceptance checklist and save the results
2. Trigger `android-debug`, `android-release`, and `android-pr-check` on GitHub
3. Capture workflow logs, artifact download evidence, and the failure-recovery path
