# 交互标准

[English Version](./interaction-standards.en.md)

## 适用范围

本文档定义应用内核心页面的交互标准，当前首批固化的是照片页扫描入口。后续如有同类流程，默认复用此标准，不再各页自定义。

## 扫描入口标准

### 1. 入口区块必须克制

1. 授权说明、待扫描说明使用辅助层级字体，不得压过媒体展示区。
2. 顶部入口区块应尽量低高度，优先给结果列表和媒体预览留空间。
3. 主 CTA 可以明确，但不能做成大面积 hero 卡片。

### 2. 授权后必须先说明扫描范围并直出可用结果

1. 用户授权后，页面默认要展示本次扫描范围摘要。
2. 首版默认扫描范围为最近 `360` 个媒体。
3. 该值必须来自真值源，禁止直接把数字硬编码在页面 JSX 文案中。
4. 当扫描范围未来支持配置时，入口区块只更新真值展示，不改交互结构。
5. 默认授权态不再展示“默认扫描说明”类长文案，优先只保留范围摘要本身。
6. 顶部入口只保留 `已选择 N 个媒体` 这类总览摘要，不再单独承载 `全部（N）` 风格的数量标签。
7. `全部`、`照片`、`视频` 的数量承载必须统一落在下方 tab 切换上；默认态与完成态都遵守同一承载位置，不得一会儿在头部、一会儿在底部。数量表现优先使用 `全部 5 / 照片 3 / 视频 2` 这类紧凑形式，而不是额外括号样式。
8. `全部`、`照片`、`视频` 的 tab 文字层级必须统一为紧凑移动端规格：图标 `16`、标签 `14`、数量 `12`；数量必须作为次级信息单独承载，不能再把整段 tab 文案做成同一字号。
9. 默认授权后就必须直接显示已授权媒体，而不是停留在空白页、说明态或占位态。
10. 下方 tab 在默认态就必须真实可用，至少包含 `全部`、`照片`、`视频` 三类，并能立刻切换到对应媒体结果，禁止出现“tab 可点但列表为空壳”的伪交互。
11. 在未开始扫描前，tab 数量展示当前授权范围内的真实计数；一旦开始扫描，`全部 / 照片 / 视频` 数量必须随列表中仍保留的媒体实时更新，而不是始终钉死在初始 `360` 范围上。

### 3. 扫描反馈必须页内嵌

1. 扫描进度默认采用页内嵌状态轨道，不允许使用阻断式弹窗作为主反馈。
2. 扫描入口、扫描中、扫描完成三种状态必须尽量复用同一张任务卡，不得把“入口说明卡”和“进度卡”拆成两张互不相干的容器。
3. 状态轨道必须包含：
   - 状态标题
   - 一条渐进呼吸的流水线式进度条
   - `current/total` 计数
4. 扫描中仍需保留用户对主页面的连续感知。
5. 扫描中的照片文件名或逐张文本提示默认不展示，避免干扰主流程与占用空间。
6. 扫描到正常媒体时，其反馈应自然渐隐并让位给主内容，不应出现突兀闪烁、骤停或机械式硬切。
7. 扫描结果必须以流式方式渐进呈现：已确认正常的媒体应在扫描过程中逐步从列表中消失，不得等到最后一刻才整体跳成结果页。
8. 随着正常媒体渐进消失，`全部 / 照片 / 视频` tab 的数量也必须实时同步递减，保证列表和数量口径始终一致。
9. 当扫描中的列表结果仍在持续收敛时，可在列表区域叠加轻量半透明 loading 遮罩，用来表达“数据还在变化”；遮罩必须保留底层媒体可见性，不能把页面整体盖死。

### 4. 完成态必须原位收口

1. 扫描完成后，在原入口区块显示识别出的异常媒体数量。
2. 完成态不得再弹出额外完成弹层。
3. 用户完成扫描后应能立刻继续查看下方媒体结果并决定是否再次扫描。
4. 扫描完成后，范围构成信息继续由下方筛选 tab 承载，例如：`全部（360）`、`照片（339）`、`视频（21）`；完成态不得把这组数量重新搬回头部。
5. 完成态顶部不再重复承载完整的范围构成展示，避免与下方筛选信息重复。
6. “发现异常媒体”提示应直接出现在“扫描完成”与 `progress/total` 之间，并使用异常提示色。
7. 异常提示色必须适配深色 / 浅色主题，不得使用固定单一颜色值。

### 5. 布局安全区与结果缓存必须稳定

1. 首页照片页必须适配刘海屏、打孔屏等特殊屏幕，顶部入口区与下方 tab 切换区都必须避开危险区。
2. 安全区适配不仅覆盖顶部标题，也必须覆盖 tab 区、筛选承载区与首屏可触达操作，避免横屏或异形屏下被遮挡、顶住或误触。
3. 设置页也必须适配刘海屏、打孔屏与横屏安全区，标题、section 容器与页脚都应吃进 `top / left / right / bottom` 安全区，而不是只补底部。
4. 已扫描结果必须缓存；用户下次进入页面时，应优先直出缓存结果与对应 tab 数量，不要求再次完整扫描。
5. 重扫是显式刷新动作；只有用户主动重扫或缓存失效时，才重新完整扫描并刷新缓存。
6. 缓存命中时，页面仍应保持 tab 可切换、结果可浏览，而不是先回退成未扫描说明态。

### 6. 媒体清晰度、选中、详情与重复组处理必须明确

1. 网格与详情中的媒体必须清晰可辨，默认优先展示足够清晰的原图或高保真本地源，不得把分析用缩略图、过度压缩图或故意柔化的遮罩态当作主展示态。
2. 列表缩略图的清晰度必须尽量接近系统相册观感；同一媒体在列表中的展示，不得与系统相册存在明显可感知的锐度落差。
3. 若出于性能原因需要先显示低清占位，必须在极短时间内无感升级为清晰内容，且不可让用户依赖模糊态判断“该不该删”。
4. 媒体选中态只保留选中图标本身，不得再通过整体降透明、厚重遮罩、额外提示文案或重复高亮框继续压低媒体可见性。
5. 一旦进入选中模式，所有媒体卡片右上角都应显示统一的选择指示器：未选中时为空心圈，已选中时为主色实心圆加对勾，形式应接近系统附件选择体验。
6. 媒体选中态必须使用明确、稳定、可识别的图标承载，不能以临时字符、纯文字或随意占位符替代。
7. 选中图标优先复用项目现有的 iconfont / icon 系统；若项目当前没有 iconfont，也应接入统一图标体系，而不是由单页各自拼凑符号。
8. 列表、底部 tab、详情关闭、左右切换、回收站空态、设置勾选等核心可视图标都必须使用统一的常用图标体系，不得混用 emoji、纯文本箭头、临时符号与系统字形。
9. 默认非选中模式下，网格缩略图不应直接叠加异常 tag 文案；列表层只保留必要的媒体内容、视频标识和重复数量等轻量语义，标签说明留给详情页。
10. 媒体网格默认处于非选中模式时，单击必须进入详情；查看详情不得被隐藏为二级入口，也不得要求先进入选中模式。
11. 长按媒体必须进入选中模式，并立即选中当前媒体；长按动作不得继续串到详情打开。
12. 一旦进入选中模式，单击媒体必须切换选中 / 取消选中，不得再打开详情；当选中项清空后，应自动回到非选中模式。
13. 批量选中后的底部操作必须同时提供 `清理` 与 `保留`，并以两枚并列的大按钮承载，数量直接显示在按钮文案中，例如 `清理 (1)`、`保留 (1)`。
14. 批量底栏应贴近移动端附件选择体验：红色危险按钮与绿色保留按钮等宽并列、圆角充分、固定吸附底部，不再复用详情中的双态 switch 结构。
15. 简化后的详情 viewer 默认结构固定为：顶部左侧页码、顶部右侧关闭、中部媒体主舞台、悬浮于底部的标签与操作区、最底部分页点；不得重新回退为多块信息面板堆叠的复杂布局。
16. 详情必须采用左右切换的媒体主舞台语义；每个切换项都应让媒体本体占主导，并在底部就近显示对应标签，避免用户在媒体、标签、操作之间来回找信息。
17. 左右切换的作用域默认是“当前媒体的相关集合”，即当前图命中的重复组、相似组或其他明确相关候选；若当前媒体没有相关集合，则详情保持单图查看，不得串到整页结果列表。
18. 详情首屏必须直接显示异常标签，例如 `重复`、`相似`、`低质量`、`模糊`、`曝光过度` 等；标签不可埋入折叠区、二级页或需要额外点击才能看见的位置。
19. 详情中的媒体主舞台必须优先保证“看得清、判得准”；图片宽度应尽量贴近可用宽度上限，优先做到接近 `100%` 主舞台宽度，而不是为标签或按钮预留过多空白。
20. 详情页必须适配刘海屏、打孔屏等特殊屏幕，顶部页码区、关闭按钮、左右切换按钮、底部标签区、底部操作区与分页点都必须避开危险区，保证横竖屏下都不被遮挡或误触。
21. 详情中必须新增 `保留` 操作，其语义是“误报 / 不清理”，不是删除、不是暂缓、也不是返回；用户执行后，该媒体应从本次待清理决策中被明确保留。
22. 若媒体属于重复组，详情中应通过左右切换直接浏览同组候选项，并允许用户在当前焦点媒体上直接执行 `清除` 或 `保留`，而不是强制用户进入额外的复杂子流程。
23. 若用户未进入详情、直接在网格执行重复组清理，系统默认保留最高清、质量最好的媒体，并将其他重复项作为默认清理候选；此默认策略必须在交互上保持可预期。
24. 详情中的 tag 必须使用最小可读字号，并始终限制为单行最新 `3` 个；标签要保持辅助层级，不得因换行或堆叠侵占媒体舞台。
25. `清除 / 保留` 或 `恢复 / 删除` 必须合并为底部单一 switch 式操作组件，字体同样使用最小可读字号，避免大按钮破坏简化视觉。
26. 建议态必须由系统根据当前媒体自行给出：高置信异常或非代表副本默认高亮 `清理`，代表副本或不确定项默认高亮 `保留`；但用户始终可以反向点击覆盖建议。
27. 若详情中只有 `1` 张媒体，则不得显示上一个 / 下一个按钮；若为重复组浏览，则到达第一个时隐藏上一个，到达最后一个时隐藏下一个，禁止首尾循环。
28. 扫描完成后的网格中，若某条媒体属于重复组，应直接在缩略图上显示重复数量徽标；徽标仅显示数量，颜色需与 `重复` tag 的提示色保持一致。
29. 回收站页必须沿用与照片页相同的网格交互语义：默认单击看详情、长按进入选中模式、选中模式下单击切换选中、空态退出选中模式。
30. 详情底部层级必须遵守“标签轻、开关稳、分页弱”：标签只负责说明原因，动作 switch 负责决策，分页点只承担位置提示，三者不得同时争抢主注意力。

## 动画标准

1. 扫描反馈动画必须丝滑、连续，禁止出现顿挫、抽动、闪烁。
2. 流水线进度动画应呈自然流动感，避免机械节拍、等距闪动、突兀循环重启或僵硬线性推进。
3. 流水滑动、呼吸起伏与状态切换应形成连续动势，不得因单次进度更新而突变。
4. 正常媒体对应的瞬时扫描反馈应以渐隐方式退场，让用户感知“已处理”而非“被硬切掉”。
5. 状态切换优先使用渐进显隐与平滑过渡，不使用突兀弹入弹出。
6. 动画必须服从主题与层级，不得因追求动效牺牲信息清晰度。
7. 详情进入、返回、重复组展开 / 收起以及选中反馈，都必须保持丝滑、连续，避免跳帧、闪切或层级错乱。
8. 选中图标的显隐与状态切换应使用轻量但明确的过渡，不得出现瞬时闪变或不一致的反馈延迟。
9. 重复组展开动画必须强化空间连续感，让用户明确知道“展开的是同一组媒体”，而不是误判为切换到了另一块内容。
10. 详情主舞台的左右切换必须像同一条媒体轨道上的连续滑动，禁止出现整屏硬切、先糊后清、瞬时跳位或标签与媒体不同步的问题。
11. 媒体从网格进入详情、以及详情内左右切换时，应优先保持画面主体清晰；允许轻量过渡，但不得用长时间模糊来伪装加载或制造“高级感”。
12. `保留`、`删除`、`选中` 等反馈动画必须轻量、明确、即时，不能因追求动效而拖慢用户判断或遮住媒体内容。
13. 列表缩略图、详情主舞台和左右翻页之间的媒体清晰度必须保持稳定，不得出现列表清楚、点开变糊，或左右翻页时先糊后清的反直觉落差。
14. 所有高频可触达控件都应具备轻量的 iOS 式按压反馈，例如细微缩放、透明度与底色变化；禁止粗糙的瞬时闪白、重型涟漪或完全无反馈。

## 实现纪律

1. 文案来自统一国际化真值源。
2. 主题色来自统一主题真值源。
3. 交互标准更新时，必须同步补测试。
