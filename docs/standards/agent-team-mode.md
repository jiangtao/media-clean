# 团队模式运行标准

[English Version](./agent-team-mode.en.md)

## 适用范围

本文档定义“团队模式”在本仓库中的实际运行方式。它不是目标文件里的静态人物介绍，而是 Lead 何时必须启用 subAgent、如何分工、怎样汇报、何时复核、何时继续推进的运行规则。

与 [执行规范](./execution-standards.md) 的关系是：

1. `execution-standards.md` 规定长期纪律。
2. 本文规定团队模式如何真正跑起来。

## 一、什么叫团队模式

团队模式成立，至少同时满足以下条件：

1. 主线程始终担任 **包拯 / Lead**。
2. 当前波次被拆成 2 个及以上相互独立的工作包。
3. 每个工作包都有明确 owner、文件范围、验证命令、完成定义。
4. 子成员汇报必须携带证据，不能只汇报“做完了”。
5. 当前波次有剩余明确 TODO 时，不因一个局部 green checkpoint 停止推进。

只要缺任何一条，就不算真正的团队模式。

## 二、默认角色映射

| 角色 | 默认承担者 | 职责 |
|------|------------|------|
| 包拯 | 主线程 | 锁定波次、拆工作包、放行/冻结、决定是否继续推进 |
| 公孙策 | `explorer` | 负责架构解释权、边界、真值源、能力分层 |
| 展昭 | 主线程或 `worker` | 负责主路径实现与跨模块收口 |
| 张龙 / 赵虎 / 王朝 / 马汉 | `worker` | 负责边界清晰、文件范围明确的实现或测试工作包 |
| 八贤王 | 主线程或审查型 agent | 负责验收、挑错、拒绝无证据完成 |

## 三、何时必须开启团队模式

满足任意一组条件，就默认开启团队模式：

1. 当前波次至少有 2 个 `P0/P1` 工作包，且文件范围可拆开。
2. 一个工作包在实现，另一个工作包可并行做测试、验证或架构核查。
3. 主线程如果不并行，就会被纯调研、纯文档、纯验证拖慢主路径实现。

以下情况不开：

1. 下一步高度耦合，所有动作都依赖同一个即时结果。
2. 只剩一个很小的局部改动。
3. 并行后会明显引入写冲突。

## 四、工作包最小格式

每个工作包都必须具备：

1. `owner`
2. `目标`
3. `写文件范围`
4. `可读上下文`
5. `验证命令`
6. `完成定义`
7. `未解决依赖`

推荐模板：

```md
### Work Packet: <slug>
- Owner: <角色/成员>
- Goal: <一句话>
- Write Scope:
  - <file-or-dir>
- Read Context:
  - <file-or-dir>
- Verification:
  - <command>
- Done When:
  - <explicit outcome>
- Blockers:
  - <if any>
```

## 五、汇报格式

所有子成员汇报统一为四段：

1. `改动范围`
2. `验证证据`
3. `未收口风险`
4. `建议下一步`

若没有验证证据，Lead 不得接受该工作包完成。

## 六、复核顺序

团队模式下默认复核顺序：

1. **规格复核**：是否对齐目标、标准、BDD、工作包要求。
2. **代码质量复核**：是否引入回归、耦合、未测路径、伪完成。
3. **终验**：是否达到当前波次完成定义。

规格未过，不进入代码质量复核。

## 七、提速规则

1. 文档、实现、验证可以并行时，必须并行。
2. 主线程不等待非关键路径调研结果，除非下一步被其阻塞。
3. 同一工作包若连续 2 轮修复仍未收敛，立即升到公孙策裁决。
4. runtime 敏感改动必须在同波次补回归，不允许“先改再说”。
5. 若当前波次已有明确剩余 TODO，八贤王的 green 只代表“当前包通过”，不代表整个波次停止。

## 八、当前仓库的默认团队模式

以当前 `v0.3` 波次，默认运行如下：

1. 包拯：主线程，负责 `v0.3` 波次调度与子成员派工。
2. 公孙策：专管产品化流程边界、清理报告真值与 observability foundation 口径。
3. 展昭：主路径实现，优先做跨模块集成与阻断裁决。
4. 执行小队：
   - 张龙：首页五步流程与产品化入口
   - 赵虎：扫描页识别摘要与状态接线
   - 王朝：保留和清理页累计报告与 `SQLite` 聚合
   - 马汉：observability foundation 与文档忽略规则
5. 八贤王：每波次终验，并维护“未收口风险”。
