# 识别召回与设计细化 - 设计总览

[English Version](./_index.en.md)

## 背景

当前仓库已经具备本地扫描、重复识别、异常识别、回收站、详情查看与基础缓存能力，但最近一轮用户反馈说明两个核心问题仍未收口：

1. 识别侧仍偏保守，模糊、纯色、灰暗、低质量、相似图以及部分完全重复图存在明显漏检。
2. 设计侧已完成主路径搭建，但扫描卡、详情舞台、标签、badge、回收站语义与动效节奏还不够“简洁、利落、可信、本地化、流畅”。

本设计用于组织下一轮正式执行波次，使团队能够在不背离 [v0.1 目标](/Users/jt/places/personal/app-cleaner/docs/goal/v0.1.md) 的前提下，同时推进“识别召回率提升”和“移动端设计 refinement”。

## 设计目标

1. 提高坏图与重复/相似图的召回率，尤其覆盖：
   - 模糊、灰暗、纯色、低质量图片
   - 两张完全相同但当前未被识别的图片
   - 相似但不完全重复的图片
2. 保持当前“本地优先、可解释、安全清理”的产品基线，不引入云端模型依赖。
3. 保持扫描性能可控，不得因提升召回率而破坏主线程流畅性与 runtime 零错误门禁。
4. 以“简洁、利落、可信、本地化、流畅”为统一风格，细化扫描页、详情页、badge/tag/回收站语义与动画。

## 推荐方案

采用双轨并行、分层收口的方案：

1. **识别轨**：建立 `Recognition Recall V2`
   - 放宽坏图识别阈值，增加中度异常层
   - 新增独立 `blur score`
   - 为重复识别增加 fallback 重试、双层指纹与相似分层
   - 把“组总数”和“待处理数量”拆开，减少用户对“没识别出来”的误判
2. **设计轨**：建立 `Mobile Refinement V2`
   - 合并扫描入口卡、进度条和完成态
   - 收紧详情页层级与悬浮动作区
   - 统一 badge/tag/回收站语义
   - 以单一主过渡收顺动效，降低扫描过程闪烁

## 不做事项

1. 本轮不引入云端 AI 模型。
2. 本轮不把全量扫描默认范围直接改成全库扫描，但会设计“扩展扫描范围”的能力。
3. 本轮不更换跨平台技术栈，不引入新的原生 worker 方案；扫描让渡仍以现有 `cooperative-yield` 为基础。

## 团队执行建议

1. **包拯**：冻结范围、裁决波次、放行门禁。
2. **公孙策**：负责识别架构与真值源，防止阈值散落。
3. **展昭**：负责识别链与 UI 主路径的最终集成。
4. **张龙 / 赵虎**：先补 BDD 失败测试，再补识别实现。
5. **王朝 / 马汉**：承担设计 refinement 与交互回归。
6. **designer**：把品牌词沉到组件与页面细节，避免只停留在口头风格。
7. **八贤王**：对照质量闸与体验验收标准做终验。

## 成功标准

1. 相同照片、近重复照片、相似照片的漏检率明显下降，并有失败样本测试覆盖。
2. 模糊、纯色、灰暗、低质量图片能稳定进入候选结果，而不是只抓极端坏图。
3. 列表与详情中的数量语义一致，用户能清楚理解“组数量”和“待处理数量”。
4. 扫描页与详情页视觉风格更统一，动效更稳，不新增 runtime / 卡顿 / 红屏。
5. 通过 `npm run typecheck -- --pretty false` 与 `npm run test -- --run`。
