# 质量优先执行体系设计

[English Version](./_index.en.md)

## 背景

依据 [docs/goal/v0.1.md](../../goal/v0.1.md) 的团队设定与“执行过程非必要不要停”的原则，当前仓库已经具备多条设计、计划、标准与实现链路，但“团队如何参与后续执行”与“runtime 报错时如何冻结/放行 TODO”仍缺统一成文约束。

这会带来三类问题：

1. 团队角色停留在目标文件中，未真正沉到现行执行规范。
2. 质量第一虽被反复强调，但 runtime 调整后“何为无 error、何时可继续 TODO”没有硬门槛。
3. TODO 虽多，推进顺序与阻断条件未被统一，容易出现体验优化压过交付阻断项。

## 目标

1. 将包拯、公孙策、展昭、执行小队、八贤王的职责固化为可执行规范。
2. 将“客户端质量第一、runtime 报错先清零”固化为质量闸。
3. 将后续 TODO 统一为可排序、可验收、可冻结的推进队列。
4. 保证新规范与现有 live 路径、BDD 场景、测试能力一致，不另起一套遗留流程。

## 方案选择

### 方案 A：继续靠目标文件与会话约定维持团队执行

- 优点：零额外文档成本。
- 缺点：团队角色只在目标文件中存在，无法直接约束后续实现与验收。

### 方案 B：新增执行规范文档，并配套设计/计划把团队与质量闸沉到仓库

- 优点：最小增量即可把团队角色、质量闸、TODO 队列和验收源显式化。
- 缺点：需要维护一份长期标准，并要求后续执行同步更新。

### 方案 C：直接做自动化流程脚本，不先成文

- 优点：一旦落成，执行成本低。
- 缺点：当前仓库尚无统一 lint/CI 套路，若不先成文，自动化会缺少稳定的规则源。

## 决策

采用 **方案 B**。

理由：

1. 这是把目标文件中“团队设定”变成仓库内真实执行纪律的最小闭环。
2. 当前仓库已有标准文档、设计文档、计划文档基础，新增一份执行规范即可自然融入。
3. runtime 与 TODO 队列本质上先是规则问题，再是自动化问题；先成文，再自动化。

## 详细设计

### 1. 执行规范成为长期真值源

- 新增 [execution-standards.md](../../standards/execution-standards.md) 作为长期执行标准。
- 该标准不取代 `v0.1`，而是负责把 `v0.1` 中的团队设定和规则具体化。

### 2. 团队角色落地到 subAgent 工作流

- 包拯：主线程 Lead，控制波次、裁决取舍、冻结或放行 TODO。
- 公孙策：架构型 subAgent 或主线程架构审查。
- 展昭：主路径集成与关键实现负责人。
- 张龙、赵虎、王朝、马汉：可并行子任务 worker，必须有明确文件所有权。
- 八贤王：验收者，要求结果同时对齐目标、标准、设计、BDD 与测试。

### 3. runtime 质量闸前置

- 任何 runtime 敏感改动都必须先通过 `typecheck` 与 `vitest --run`。
- runtime 一旦报错，非 P0 TODO 冻结。
- “无 error”必须由测试、类型检查与 live 路径验证共同定义，不能只凭人工感觉。

### 4. TODO 队列显式分级

- `P0`：runtime、崩溃、主路径阻断。
- `P1`：目标文件要求但 live 路径未贯通。
- `P2`：交互优化与体验细化。
- `P3`：远期扩展与预研。
- 每个 TODO 必须带负责人、依赖、BDD 场景、验证命令与完成定义。

### 5. 当前最重要的交付阻断项

- 回收站页真实数据加载
- Tab badge 真实计数
- 回收站详情链路

上述事项在规范中被定义为当前默认阻断 TODO，不应被体验细节掩盖。

## 风险与边界

1. 本设计不修改 [docs/goal/v0.1.md](../../goal/v0.1.md)。
2. 本设计先解决“执行规范与优先级”，不直接等同于一次性做完所有 TODO。
3. 当前仓库没有统一 lint/CI，故质量闸先以 `typecheck + vitest --run` 为最低闭环。
