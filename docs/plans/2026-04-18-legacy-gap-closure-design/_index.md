# Claude 遗留缺口收口设计

## 背景

依据 [docs/goal/v0.1.md](../goal/v0.1.md) 的不可变目标，当前仓库已经具备 Android 优先 MVP 的主体能力，但仍存在两类未收口问题：

1. `docs/reports/claude-mem-screen-adaptation-debug.md` 记录的特殊屏适配调试尚未完全结束，`Samsung_S20` 横屏左上打孔识别仍残留失败。
2. 用户人工反馈确认照片页、回收站页尚未完整接入全局国际化与多主题，说明现有“页面各自读取设置”的方式没有真正做到 App 级一致。

本设计不修改目标文件，只负责将遗留缺口收口到“可验证、可持续推进”的状态。

## 目标

1. 建立以包拯为 Lead 的执行团队，并把职责固化到设计和计划中。
2. 统一 App 级语言/主题真值源，禁止页面继续各自读存储、各自判主题。
3. 补齐照片页、回收站页及其相关组件的国际化和多主题覆盖。
4. 优先完成用户目标机型的特殊屏适配，本轮默认以仓库中唯一明确残留的 `Samsung_S20` 为目标机型。
5. 为 App 入口、主题、国际化、特殊屏适配补足测试和验证闭环。

## 方案选择

### 方案 A：页面各自读取存储并局部修补

- 优点：改动小，短期见效快。
- 缺点：状态源分散，语言/主题切换存在不同步风险；后续继续补页面会不断复制逻辑。

### 方案 B：建立轻量 AppPreferences 根上下文并收口到 App 入口

- 优点：语言、主题、文案、状态栏样式统一收口；导航、页面、组件共享同一份真值；适合后续继续扩展。
- 缺点：需要调整 `App.tsx`、导航与页面的依赖注入方式。

### 方案 C：引入外部状态管理库

- 优点：可扩展性强。
- 缺点：当前需求过小，属于过度设计，会扩大遗留收口范围。

## 决策

采用 **方案 B**。

理由：

1. 这是最小可持续方案，既能补当前缺口，也能把 App 类约定真正固定在入口。
2. 主题/语言覆盖问题本质是“真值源分散”，不是某一两个页面漏翻译这么简单。
3. 特殊屏适配不只是检测函数，还包括页面顶底部留白和动作栏位置；统一上下文后更容易让页面遵守约定。

## 详细设计

### 1. App 入口约定

- `App.tsx` 只负责根级 Provider 装配：`SafeAreaProvider`、`AppPreferencesProvider`、`NavigationContainer`、`StatusBar`。
- 所有页面和导航禁止直接从存储层读取语言/主题偏好。
- `SettingsScreen` 只能通过上下文更新偏好，再由上下文负责持久化。

### 2. AppPreferences 根上下文

- 提供 `language`、`themePreference`、`resolvedThemeScheme`、`theme`、`copy`、`setLanguage`、`setThemePreference`。
- 启动时一次性加载 `app-language-storage` 与 `theme-preference-storage`。
- 统一处理系统主题跟随逻辑。

### 3. 页面与组件收口

- `MainTabNavigator`、`PhotoGridScreen`、`RecycleBinScreen`、`SettingsScreen` 全部改为消费上下文。
- `SegmentedControl`、`PhotoGrid` 等直接出现在照片页/回收站页的组件补主题色，不再使用硬编码浅色常量。
- `PhotoGridScreen` 与 `RecycleBinScreen` 增加顶部安全区意识，避免刘海/打孔横屏时内容顶到危险区。

### 4. 特殊屏适配

- 对 `detectHolePunch` 的左上打孔判断与调试报告对齐：
  - 横屏
  - `left > right`
  - `left >= 26`
- 页面布局层同步验证：
  - 顶部筛选区域/头部需避开顶部与横屏左侧危险区
  - 底部操作栏继续避开底部手势区

### 5. 验证闭环

- 先补 RED：新增/更新 BDD 测试覆盖 App 偏好上下文、页面级 i18n/主题、`Samsung_S20` 左上打孔识别。
- 再做 GREEN：最小实现通过。
- 最后做 repo 级验证：相关测试、类型检查。

## 团队编组

严格遵循目标文档中的团队规则：

| 角色 | 成员 | 职责 |
|------|------|------|
| Lead | 包拯 | 总控范围、裁决取舍、监督闭环；本轮由主线程承担 |
| 架构 | 公孙策 | 审核 App 入口约定、上下文边界、跨模块一致性 |
| 核心执行 | 展昭 | 核心实现与集成收口 |
| 执行小队 | 张龙、赵虎、王朝、马汉 | 页面/组件/i18n/兼容性测试分工执行 |
| 验收 | 八贤王 | 以目标文档和 BDD 为准验收 |

## 风险与约束

1. 工作区当前是脏的，且 `docs/goal/v0.1.md` 已有外部改动；本轮不得覆盖这些改动。
2. 目前没有更明确的用户机型证据，因此默认以 `Samsung_S20` 作为目标机型假设推进。
3. 现有部分测试以“逻辑占位”形式存在，本轮优先补关键行为闭环，不追求一次性重写全部测试。
