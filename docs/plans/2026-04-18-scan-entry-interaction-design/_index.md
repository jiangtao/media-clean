# 扫描入口交互优化设计

## 背景

当前照片页扫描入口已经具备权限检查、授权引导和扫描进度能力，但交互形态仍然偏“拦截式”：

1. 授权卡片与扫描 CTA 字号偏大，占用了本应留给媒体展示区的空间。
2. 扫描进度通过弹窗展示，会中断用户对主页面的连续感知。
3. 扫描完成后依赖弹层完成态，而不是把“发现了多少异常媒体”留在页面主叙事中。
4. 授权后的默认态没有明确说明本次扫描的默认范围，用户无法立刻建立“将扫描多少媒体”的预期。

## 目标

1. 将扫描入口改为页内嵌式交互，不再使用弹窗承载扫描过程。
2. 授权态与待扫描态采用更克制的文字层级，把垂直空间让给媒体展示区。
3. 扫描中使用“一条渐进呼吸的流水线”表达进度，明确展示 `current/total`。
4. 扫描完成后在同一区块展示识别出的异常媒体数量，而不是弹窗收尾。
5. 将该交互沉淀为可复用的交互标准，后续页面和重构必须遵守。

## 方案比较

### 方案 A：保留卡片 + 弹窗，只压缩字号

- 优点：改动最小。
- 缺点：交互核心问题未解决，扫描仍会打断主页面。

### 方案 B：改为页内扫描轨道，统一待扫描 / 扫描中 / 完成态

- 优点：主页面连续、空间利用更高、可直接承载扫描范围与结果摘要。
- 缺点：需要调整 `PhotoGridScreen` 与 `ScanProgress` 的职责边界。

### 方案 C：做完整 hero 仪表盘

- 优点：视觉表现力强。
- 缺点：超出当前 MVP 所需，容易侵蚀展示区并扩大改动范围。

## 决策

采用 **方案 B**。

理由：

1. 它解决的是交互结构问题，而不是样式表面问题。
2. 它与“把空间留给展示区”的目标一致。
3. 它能自然承载“默认扫描最近 360 个媒体”这一范围摘要，以及扫描完成后的异常数量摘要。

## 设计摘要

1. 照片页顶部保留一个低高度的入口区块，承载权限说明、扫描范围摘要和主 CTA。
2. `ScanProgress` 从弹窗改为页内内嵌状态轨道组件。
3. 状态轨道包含：
   - 小号状态标题
   - 一条渐进呼吸的流水线式进度条
   - `current/total` 计数
   - 扫描中的当前文件名
   - 扫描完成后的异常媒体数量
4. 默认授权态展示本次扫描范围摘要，首版默认从真实常量 `DEFAULT_SCAN_LIMIT` 读取 `360`。
5. 所有规则同步写入独立交互标准文档，作为后续实现和验收依据。

## 风险与边界

1. 本轮只收口照片页扫描入口，不顺手扩大到其他页面。
2. “可改的 360”本轮先做成可扩展真值源展示，不在本轮新增完整扫描条目数设置能力。
3. 扫描取消仍沿用当前能力边界，不在本轮补底层可中断扫描。
