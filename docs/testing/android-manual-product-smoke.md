| 字段 | 内容 |
| --- | --- |

| 用例名称 | Android 产品主路径与回收站安全闭环人工冒烟 |
| 用例类型 | 手工 smoke / 真机验收 |
| 优先级 | P0 |

## 产品交互目标

本用例用来确认当前 App 的核心用户体验仍然成立：

1. 用户打开 App 后能稳定进入权限、相册或识别状态，不被重复骨架屏打断。
2. 相册识别结果能进入列表和详情，详情页支持按当前结果连续左右浏览。
3. 用户能把候选图片移入应用内回收站，并在回收站内恢复或永久删除。
4. 回收站 badge、列表、详情索引和数据状态保持一致。
5. 所有关键页面在小屏真机上没有遮挡、错位、白屏、重复 loading 或用户不可理解的中间态。

## 前置条件

1. Android 真机已连接并保持解锁。
2. `adb devices -l` 能看到目标设备处于 `device` 状态。
3. App 已安装当前待验收构建。
4. 若使用 debug / dev-client 包，Metro 已启动并且设备可访问 Metro。
5. 真机相册中准备至少一组可识别样例，建议包含：
   - 2 张以上相似或重复图片。
   - 1 张截图。
   - 1 张模糊或低质量图片。
   - 1 个非重复的普通图片样例。

推荐命令入口：

```bash
npm run build:android:debug
npm run observability:device:doctor -- --serial <android-serial>
npm run verify:android:observability -- --serial <android-serial>
```

## 执行步骤

| 步骤 | 操作 | 预期结果 |
| --- | --- | --- |
| 1 | 清空 App 数据或使用干净安装包启动 App | App 能冷启动，不出现原生崩溃、白屏或停留在系统桌面 |
| 2 | 首次进入 App，观察权限/相册初始状态 | 只在权限或媒体库基础状态未确定时出现骨架屏；不连续闪 2 到 3 轮骨架 |
| 3 | 拒绝相册权限 | 出现可理解的权限空态或授权引导；不循环 loading，不崩溃 |
| 4 | 重新允许相册权限 | 进入已授权相册状态；授权后不再额外闪一轮权限骨架 |
| 5 | 观察相册加载过程 | 骨架屏只表现为整体渐进效果；卡片没有明显离开动画 |
| 6 | 进入识别前状态 | 页面表现为可操作的业务状态，而不是再次显示骨架屏 |
| 7 | 发起识别或等待识别进行 | 展示识别中进度、计数或处理状态；不把识别中误表现为骨架屏 |
| 8 | 等待识别完成并进入结果态 | 结果稳定展示；识别完成后不再出现多轮中间骨架闪烁 |
| 9 | 在结果列表点击一个分组或图片 | 能进入图片列表或详情页；图片、标题、计数与入口一致 |
| 10 | 在结果详情页左右滑动 | 可以按当前结果列表连续切换；第一页和最后一页边界正确；索引不跳、不倒退 |
| 11 | 返回结果列表，选择若干候选并移入回收站 | 选择态、数量、操作反馈和回收站 badge 同步更新 |
| 12 | 查看底部或右上角回收站 badge | 图标尺寸不过大；数字可见、居中；1 位、2 位和 `99+` 不挤压、不裁切 |
| 13 | 进入回收站列表 | 显示所有已回收项目；列表可滚动；空态和非空态切换正确 |
| 14 | 点击回收站任一图片进入详情 | 详情页打开成功；当前图片、索引和回收站列表位置一致 |
| 15 | 在回收站详情页连续左右滑动 | 浏览范围覆盖整个回收站列表，不只限制在某个重复组内 |
| 16 | 在回收站详情页恢复当前图片 | 当前项移出回收站；badge 和列表数量减少；详情落到下一张或返回合理列表状态 |
| 17 | 在回收站详情页永久删除当前图片 | 有明确反馈；当前项从回收站移除；badge、列表和详情索引同步 |
| 18 | 从详情返回回收站，再返回照片页 | 页面状态稳定；不白屏、不重复识别、不丢失刚才的回收站决策 |
| 19 | 切后台再回前台 | 当前页面和关键状态尽量保持；不回到错误入口、不重置为骨架屏 |
| 20 | 打开设置页并返回照片页 | 设置页可打开；返回后照片页、回收站 badge 和选中态没有异常回退 |

## 通过标准

本用例通过需要同时满足：

1. 步骤 1 到 20 没有 P0 阻断问题。
2. 冷启动、授权后、识别后不出现肉眼可感知的连续骨架屏闪烁。
3. 回收站详情页可以按整个回收站列表左右滑动。
4. 回收站 badge 数字完整可见、居中，且不会因为位数变化导致明显错位。
5. 恢复和永久删除后，列表数量、详情索引、badge 数字保持一致。
6. 全流程没有白屏、崩溃、不可关闭 loading、按钮遮挡或关键文案缺失。

## 失败判定

出现以下任一情况，本用例判定失败：

1. App 启动后停留在系统桌面、白屏或 native crash。
2. 相册主路径出现 2 到 3 次以上骨架屏闪烁，影响用户判断页面状态。
3. 识别前、识别中、识别后被同一种骨架屏抹平，用户无法判断真实业务状态。
4. 回收站详情页只能在重复组内左右滑，不能覆盖整个回收站列表。
5. 回收站 badge 数字不可见、被裁切或明显不居中。
6. 恢复或永久删除后出现数量不一致、索引错误、详情卡死或回收站空态错误。

## 证据记录

每轮人工自测至少记录：

1. 设备型号、系统版本、App 包名、构建类型。
2. 执行时间和测试人。
3. 是否清空 App 数据。
4. 失败步骤的截图或录屏。
5. 失败时的 artifact 路径或 logcat 片段。

推荐 artifact 命名：

```text
artifacts/manual-smoke/android-product-smoke-YYYYMMDD-HHMM/
```

## 自动化补充信号

人工冒烟通过后，建议按改动范围补充以下命令作为证据：

```bash
npm run verify:maintainability
npm run test -- --run src/ui/screens/__tests__/PhotoGridScreen.test.tsx src/ui/screens/__tests__/RecycleBinScreen.test.tsx src/navigation/__tests__/TabBar.test.tsx
npm run verify:android:recycle -- --serial <android-serial>
npm run verify:android:recycle-delete -- --serial <android-serial>
```

自动化失败但人工通过时，需要记录失败原因；人工失败时，不得用自动化通过替代产品验收。
