# v0.0.1 Release 审计

[English Version](./v0.0.1-audit.en.md)

本文档记录 `v0.0.1` 当前 release 审计结果，目标是把“已经拿到强证据的事项”和“仍需下一轮补完的事项”明确分开，避免把构建通过误判成整体交付完成。

## 审计范围

本轮仅覆盖以下内容：

1. `v0.0.1` 版本口径是否一致
2. Android-first MVP 是否具备基础发布证据
3. Android debug / release 本地自动化链路是否真实可跑
4. 发布页 / Vercel 资源是否已经纳入仓库并可本地构建
5. 当前仍缺哪些交付证据

## 目标映射

| 目标项 | 当前状态 | 证据 |
| --- | --- | --- |
| 基础 MVP 跑通 | 已拿到强证据 | `npm run test -- --run`、`npm run typecheck`、`npm run export:android` |
| 设计稿设计好 | 部分完成 | 设计与执行方案见 `docs/plans/2026-04-16-android-media-cleaner-*`；发布页资源已入仓 |
| Android 产品完备性 check | 未完成 | 当前只有单测与本地构建证据，缺真实交互验收 |
| Android 自动化流程 check | 部分完成 | 本地 release/debug 发包脚本真实跑通；GitHub Actions 仍缺远端运行证据 |

## 产品完备性现有证据

当前“产品完备性”主要还是测试级证据，不是最终交互级验收：

1. 扫描与进度：`src/features/scan/__tests__/scan-with-progress.test.ts`
2. 回收站与候选状态：`src/application/media-cleaner-helpers.test.ts`、`src/ui/screens/__tests__/RecycleBinScreen.test.tsx`
3. 设置页扫描范围、语言、主题：`src/ui/screens/__tests__/SettingsScreen.test.tsx`
4. 本地提醒与调度：`src/services/notifications/cleanup-reminders.test.ts`
5. 扫描完成通知：`src/services/notifications/scan-completion-notifications.test.ts`
6. 刘海 / 打孔 / 折叠 / 平板适配：`src/features/compatibility/__tests__/screen-adaptation.test.ts`

这些测试说明主要逻辑和适配规则已经被编码并受回归保护，但它们不能替代真实 Android 交互验收。

## 当前缺口

以下事项仍不能判定为“已完成”：

### 1. Android 产品完备性仍缺真实验收

当前还缺以下真实交互级验证：

1. 首次权限申请与拒绝后的引导
2. 扫描主流程与结果列表联动
3. 自动清理 / 选中清理 / 回收站恢复
4. 提醒设置变更后的 UI 与系统通知联动
5. 语言切换、主题切换后的页面回流
6. 刘海屏、打孔屏、折叠屏上的真实视觉检查

### 2. GitHub Actions 仍缺远端证据

当前已经有：

1. `.github/workflows/android-release.yml`
2. `.github/workflows/android-debug.yml`
3. `.github/workflows/android-pr-check.yml`

但本轮没有拿到：

1. GitHub runner 的真实成功记录
2. `actionlint` 或等价 workflow 语法校验结果

因此目前只能说“workflow 设计存在，且本地等价脚本已跑通”，还不能说“远端自动化已经最终闭环”。

## 结论

截至当前审计时点：

1. `v0.0.1` 的基础 MVP、发布页资源、Android 本地 debug / release 发包链路都已经拿到强证据
2. `Android 产品完备性` 与 `GitHub Actions 远端自动化闭环` 仍未完成最终验收
3. 因此当前状态适合表述为：

> `v0.0.1` 已具备本地发布与审计基础，但整体 release 目标尚未完成，剩余工作需要在下一轮专门 session 中收口。

## 下一轮建议

1. 用真实设备或 Android emulator 做一轮产品流转验收，并把结果落成 checklist
2. 在 GitHub 上实际触发 `android-debug`、`android-release`、`android-pr-check`
3. 补齐 workflow 级日志、artifact 下载记录与失败回归策略
