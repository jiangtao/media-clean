# 架构设计

[English Version](./architecture.en.md)

## 1. 规则分层

执行体系分为四层：

1. **目标层**：`docs/goal/v0.1.md`
2. **标准层**：`docs/standards/execution-standards.md`
3. **设计/计划层**：当前设计与后续执行计划
4. **实现/验证层**：代码、测试、`.feature` 场景与验收命令

目标层定义“为何而做”，标准层定义“如何长期执行”，设计/计划层定义“本轮怎么落”，实现/验证层定义“是否做成”。

## 2. 团队控制流

```text
包拯(Lead)
  ├─ 公孙策(架构审查)
  ├─ 展昭(主路径集成)
  ├─ 张龙/赵虎/王朝/马汉(并行执行)
  └─ 八贤王(验收)
```

控制原则：

1. Lead 拥有最终任务拆分权与冻结权。
2. 架构审查先于跨模块合并。
3. 并行执行必须以文件边界分治。
4. 验收必须独立于实现者。

## 3. runtime 质量闸架构

runtime 相关工作采用“先止血、后扩张”策略：

1. 发现 runtime 报错
2. 当前波次冻结为 `P0`
3. 补最小失败测试或定向回归
4. 修复根因
5. 过 `typecheck`
6. 过定向测试
7. 过全量测试
8. 恢复原 TODO 波次

## 4. TODO 队列架构

当前默认队列：

1. `P0` runtime 主线  
   - 扫描缓存
   - 主线程让渡
   - 提醒冷启动 reconcile
2. `P1` 交付阻断  
   - 回收站真实数据
   - badge 真实计数
   - 回收站详情
3. `P2` 交互与体验继续收口
4. `P3` 远期扩展

## 5. 验收架构

每一项任务至少经过四类对照：

1. 目标文件对照
2. 标准对照
3. BDD 对照
4. 测试/命令对照

任一未过，任务不关闭。
